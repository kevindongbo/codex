# 东铂跨境 ERP 架构与业务规则

本文档定义东铂跨境从浏览器单机版升级为可自建、多人协作 ERP 的业务边界。现有 GitHub Pages 继续作为本机模式；当前团队后端使用同源 HTTPS、数据库、账号权限和外部图片 URL。对象存储、异步任务与前端团队模式接入属于后续阶段。

## 1. 部署形态

### 本机模式

- 入口：GitHub Pages 或直接打开静态网页。
- 数据：当前浏览器 `localStorage`。
- 适合：演示、个人试用、无服务器时临时使用。
- 限制：不跨设备共享、没有账号审计、本地图片不能共享。

### 团队模式

- Web：Caddy/Nginx 提供 HTTPS 和静态前端。
- API：Django + Django REST Framework 模块化单体。
- 数据库：PostgreSQL。
- 异步任务：当前未接入；出现真实定时任务后再增加 Redis 与任务执行器。
- 图片：当前保存外部 HTTPS URL；需要上传文件时再同时接入上传端点与 S3/MinIO。
- 部署：Docker Compose，后续可平滑迁移至 Kubernetes。

前端与 API 同源部署，例如 `https://erp.example.com/` 与 `https://erp.example.com/api/`，避免令牌跨域泄露和复杂的 CORS 配置。

## 2. 模块边界

1. **账号与权限**：组织、成员、角色、仓库/店铺数据范围。
2. **商品中心**：SPU/SKU、图片、平台链接、标准成本、供应商、启停状态。
3. **采购管理**：供应商、采购单、采购明细、预计到货、收货单。
4. **库存管理**：库存余额、不可变库存流水、盘点、调整、库位与批次扩展。
5. **订单履约**：销售订单、库存锁定、拣货、复核、出库和物流单号。
6. **退货管理**：退货申请、收货、质检结果、重新入库或报损。
7. **竞品监控**：竞品链接、公开数据快照、趋势、异常提醒。
8. **系统治理**：审计日志、幂等请求、导入导出、任务和备份恢复。

模块之间通过业务服务调用；库存数量不能由普通 CRUD 接口直接修改。

## 3. 核心业务状态

### 商品

```text
draft -> active -> inactive
```

- 草稿不能进入采购、库存调整或订单选择器。
- 已产生库存或业务单据的 SKU 不能改成竞品或退回草稿，只能停用。
- 本店 SKU 启用前必须具备唯一 SKU、商品链接、商品图片和成本。

### 采购单

```text
draft -> submitted -> partial -> received
   \-----------> cancelled
```

- 草稿不计入在途。
- 在途数量为有效采购数量减去已收和已取消数量。
- 收货必须在一个数据库事务中同时写收货单、库存流水和余额。
- 重复提交同一个收货幂等键不能重复增加库存。

### 销售订单

```text
draft -> ready -> allocated -> picking -> verified -> shipped
   \--------\----------\---------\----------> cancelled
```

- 锁定采用整单成功或整单失败，不产生无法解释的部分锁定。
- `allocated` 表示库存已锁定；仓库随后显式推进“开始拣货”和“复核完成”，只有 `verified` 才能出库。
- 出库在一个事务中同时减少已在库和锁定，并生成出库单及流水。
- 已出库订单不可取消，只能按原明细办理退货。

## 4. 库存不变量

```text
在途 = 采购有效数量 - 已收货 - 已取消
已在库 = 所有已过账库存流水的 on_hand_delta 之和
锁定 = 所有有效锁定流水的 reserved_delta 之和
可用 = 已在库 - 锁定
```

任何时刻必须满足：

- `on_hand >= 0`
- `reserved >= 0`
- `reserved <= on_hand`
- 每个仓库、SKU 只有一条当前余额记录。
- 余额可以从不可变流水完整重建。
- 业务过账接口使用数据库事务、行锁和幂等键。
- 已过账流水不可编辑或删除，纠错通过反向流水完成。

## 5. 核心数据库表

| 领域 | 核心表 | 关键约束 |
|---|---|---|
| 组织 | `organizations`、`memberships` | 成员在组织内唯一 |
| 范围 | `warehouses` | 仓库编码在组织内唯一；细粒度仓库授权待扩展 |
| 商品 | `products`、`skus`、`product_images` | SKU 在组织内唯一 |
| 采购 | `suppliers`、`purchase_orders`、`purchase_order_lines` | 单号在组织内唯一 |
| 收货 | `receipts`、`receipt_lines` | 收货数量不得超过未收数量 |
| 库存 | `stock_balances`、`stock_ledger` | 仓库 + SKU 唯一余额 |
| 订单 | `sales_orders`、`sales_order_lines`、`stock_reservations` | 订单号在组织内唯一 |
| 出库 | `shipments`、`shipment_lines` | 同一订单不可重复出库 |
| 退货 | `return_orders`、`return_lines`、`return_receipts` | 累计退货不得超过已出库 |
| 监控 | `competitor_snapshots`、`monitoring_alerts` | 商品 + 快照时间唯一 |
| 治理 | `audit_logs`、各过账单据的幂等键 | 幂等键在组织内唯一 |

所有业务表包含 `organization_id`，并使用 UUID 主键、创建时间、更新时间；业务单据额外记录创建人和最后操作人。

## 6. 权限模型

默认角色：

- **系统管理员**：成员、权限、全部业务和系统设置。
- **运营主管**：商品、采购、订单、竞品和报表，可审批调整。
- **采购员**：供应商、采购单和收货，不可修改权限。
- **仓库员**：收货、盘点、拣货、复核、出库和退货。
- **只读用户**：只读业务数据和审计日志。

当前权限同时检查动作和组织范围；仓库级成员范围是后续扩展，尚不能作为已实现能力对外承诺。

## 7. API 规则

- 统一前缀：`/api/`，所有路由保留末尾斜杠。
- 列表接口必须按当前组织隔离，禁止依赖前端传入组织 ID 作为唯一保护。
- 过账接口使用动作端点，例如：
  - `POST /api/purchase-orders/{id}/submit/`
  - `POST /api/receipts/`
  - `POST /api/stock-balances/adjust/`
  - `POST /api/orders/{id}/confirm/`
  - `POST /api/orders/{id}/allocate/`
  - `POST /api/orders/{id}/start-picking/`
  - `POST /api/orders/{id}/verify/`
  - `POST /api/orders/{id}/ship/`
  - `POST /api/orders/{id}/cancel/`
  - `POST /api/returns/{id}/receive/`
  - `POST /api/returns/{id}/reject/`
- 每个外部重试可能触发的动作在请求体中接受 `idempotency_key`；同键不同业务内容会被拒绝。
- 错误返回用户可读中文消息和字段级校验信息；稳定业务错误码仍是后续增强。
- 成本、权限、过账和状态变化必须写审计日志。

## 8. 单机数据迁移

1. 每台旧设备先导出完整 JSON；不要直接把不同设备的库存余额相加。
2. 指定一台设备为库存权威来源，逐 SKU 核对在库、锁定和在途。
3. 旧在库生成“期初库存”流水。
4. 旧在途生成“迁移待核对采购单”，确认后再转正式状态。
5. 竞品快照按商品链接与记录时间去重合并。
6. 本地 Data URL 图片需先转成可共享的 HTTPS URL，再写入商品图片表；文件上传端点尚未实现。
7. 导入完成后生成对账报告；差异不为零时禁止切换正式系统。

## 9. 验收标准

- 两个不同账号能看到同一组织数据，并按组织角色限制写操作。
- 商品、图片、快照数量与迁移报告一致。
- 在途和在库逐 SKU 对账差异为零。
- 重复收货、重复出库请求不会重复过账。
- 库存余额能从流水重建并得到相同结果。
- 成本和库存调整在审计日志中包含操作人、原因和前后值。
- 数据库与本地媒体卷完成一次真实备份和恢复演练。
- GitHub Pages 本机模式继续可用，且不会误连团队数据库。
