# 东铂跨境 ERP 架构与业务规则

本文档定义东铂跨境从浏览器单机版升级为可自建、多人协作 ERP 的业务边界。现有 GitHub Pages 继续作为本机演示模式；Docker 自建版代码已接入团队前端、同源 API、PostgreSQL、账号权限、多仓、调拨、补货建议、一键出库和外部图片 URL。仓库内有可部署配置不等于正式环境已经上线：真实服务器、域名、HTTPS、恢复演练与多人验收仍需在目标基础设施上完成。

## 1. 部署形态

### 本机模式

- 入口：GitHub Pages、公开静态站点或直接打开静态网页。
- 数据：当前浏览器 `localStorage`。
- 适合：演示、个人试用、无服务器时临时使用。
- 限制：不跨设备共享、没有账号审计、本地图片不能共享，因此不能当作联网团队版。

### 团队模式

- Web：Caddy/Nginx 提供 HTTPS 和静态前端。
- API：Django + Django REST Framework 模块化单体。
- 数据库：PostgreSQL。
- 异步任务：当前未接入；出现真实定时任务后再增加 Redis 与任务执行器。
- 图片：当前保存外部 HTTPS URL；需要上传文件时再同时接入上传端点与 S3/MinIO。
- 部署：Docker Compose；只有实际完成服务器部署与验收后才属于正式联网环境，后续可迁移至 Kubernetes。

前端与 API 同源部署，例如 `https://erp.example.com/` 与 `https://erp.example.com/api/`，避免令牌跨域泄露和复杂的 CORS 配置。

## 2. 模块边界

1. **账号与权限**：组织、成员、角色和组织数据范围。
2. **商品中心**：SPU/SKU、图片、平台链接、标准成本、供应商、启停状态。
3. **采购管理**：供应商、采购单、采购明细、预计到货、收货单。
4. **多仓库存**：自定义仓库、仓库能力、余额、不可变库存流水、调整和仓间调拨。
5. **智能补货**：采购周期、实际出库速度、库存位置、补货参数、提醒等级和可解释信心度。
6. **订单履约**：销售订单、一键事务出库，以及保留的锁库、拣货、复核、出库分步能力。
7. **退货管理**：退货申请、收货、质检结果、重新入库或报损。
8. **竞品监控**：竞品链接、完整基线快照、只录销量的增量快照、趋势和异常提醒。
9. **系统治理**：审计日志、幂等请求、导入导出、任务和备份恢复。

模块之间通过业务服务调用；库存数量不能由普通 CRUD 接口直接修改。

## 3. 核心业务状态

### 商品

```text
draft -> active -> inactive
```

- 草稿不能进入采购、库存调整或订单选择器。
- 已产生库存或业务单据的 SKU 不能改成竞品或退回草稿，只能停用。
- 本店 SKU 启用前必须具备唯一 SKU、商品链接、商品图片和成本。

### 采购单

```text
draft -> submitted -> partial -> received
   \-----------> cancelled
```

- 草稿不计入在途。
- 在途数量为有效采购数量减去已收和已取消数量。
- 收货必须在一个数据库事务中同时写收货单、库存流水和余额。
- 重复提交同一个收货幂等键不能重复增加库存。

### 销售订单

```text
draft -> ready -> allocated -> picking -> verified -> shipped
   \--------\----------\---------\----------> cancelled
```

- 普通界面使用 `POST /api/orders/{id}/confirm-and-ship/`。它在一个数据库事务内依次完成确认、整单锁库、拣货、复核和出库；任一步失败都不会留下部分锁定、部分扣减或孤立出库单。
- `allocated`、`picking`、`verified` 仍是有效底层状态；原有分步端点继续保留给需要人为分段控制的高级流程和兼容场景。
- 出库同时减少已在库和锁定，并生成出库单、明细、不可变流水与审计日志。同一个幂等键重试返回同一出库结果。
- 已出库订单不可取消，只能按原明细办理退货。

### 仓库与调拨

仓库类型包括 `overseas`（海外仓）、`forwarder`（货代仓）、`school`（学校仓）、`domestic`（国内仓）和 `other`。组织内可按实际需要创建仓库，没有写死数量；每个仓库还记录国家、时区、结构化地址/联系人、`can_receive`、`can_ship` 与 `active`。

```text
draft -> in_transit -> received
   \-----------> cancelled
        in_transit -> cancelled（撤回并退回来源仓）
```

- 发出时在同一事务中锁定调拨单和来源仓余额，逐明细扣减来源仓并写 `transfer_out` 流水；任一 SKU 库存不足则整单失败。
- 调拨在途不属于任何仓库的在库，但会计入目标仓补货计算中的“已确认在途”。
- 收货时增加目标仓余额并写 `transfer_in` 流水；已发出调拨取消时恢复来源仓并写 `transfer_cancel` 流水。
- 发出和收货分别使用幂等键；已过账调拨及其明细不可直接覆盖历史。

### 竞品快照

- 第一条快照是完整基线，至少由普通快照接口录入价格、累计销量、评分、评论数、上下架状态等已有字段。
- 后续常规更新使用 `quick-sales`，只提交竞品、累计销量和可选采集时间；服务端从最近快照继承价格、评分、评论数、上下架状态和原始字段。
- `quick-sales` 没有历史基线时会拒绝创建；每次创建都会写审计日志。系统不会把自动继承理解为重新抓取了网页。

## 4. 补货预测与提醒

预测范围始终是一个组织内的“仓库 × SKU”，不会把马来仓、货代仓和学校仓的库存或出库速度混算。

### 采购周期

1. 读取同仓库、同 SKU 近两年的采购与已完成收货记录；商品配置了默认供应商时再按该供应商筛选。
2. 优先计算从下单到完整收货的 P80，并同时返回中位数、首次收货统计、有效/排除样本数。
3. 完整收货样本至少 8 个为高信心，3–7 个为中信心，1–2 个为低信心。
4. 没有完整收货样本时使用首次收货 P80；没有任何历史时使用人工周期（默认 14 天），二者均为低信心。
5. 缺少 `ordered_at` 时以采购单创建时间近似，异常或负周期会被排除，原因在结果中明确列出。

### 出库速度

只统计实际完成的出库单，不使用订单创建量或竞品销量：

```text
加权日速度 = 7 天出库量 / 7 × 50%
             + 14 天出库量 / 14 × 30%
             + 30 天出库量 / 30 × 20%
```

出库历史覆盖至少 28 天且有至少 10 个活跃出库日为高信心；覆盖至少 13 天且有至少 3 个活跃出库日为中信心；否则为低信心。最终预测信心度取采购周期与出库速度两项中较低者。

### 库存位置与建议量

```text
可用库存 = 已在库 - 锁定
已确认在途 = 已提交/部分收货采购未收数量 + 发往本仓的调拨在途
库存位置 = 可用库存 + 已确认在途
安全库存 = max(日速度 × 安全天数, 人工安全库存)
补货点 = 日速度 ×（预测采购周期 + 检查周期）+ 安全库存
目标库存位置 = 日速度 ×（预测采购周期 + max(目标覆盖天数, 检查周期)）+ 安全库存
原始建议量 = max(0, 目标库存位置 - 当前库存位置)
建议采购量 = 原始建议量按 MOQ 和整箱数向上取整
最晚下单日 = 预计库存位置耗尽日 - 预测采购周期 - 安全天数
```

默认安全天数 7、检查周期 7、目标覆盖 30、整箱数 1；人工策略可按仓库/SKU 覆盖采购周期、检查周期、目标覆盖天数、MOQ、整箱数和安全库存。`red` 表示已触发补货，`yellow` 表示接近补货点，`green` 表示暂时健康。建议结果是可解释的决策辅助，不会自动创建或提交采购单。

## 5. 库存不变量

```text
在途 = 采购有效数量 - 已收货 - 已取消
已在库 = 所有已过账库存流水的 on_hand_delta 之和
锁定 = 所有有效锁定流水的 reserved_delta 之和
可用 = 已在库 - 锁定
调拨在途 = 已发出但目标仓尚未收货的调拨数量
```

任何时刻必须满足：

- `on_hand >= 0`
- `reserved >= 0`
- `reserved <= on_hand`
- 每个仓库、SKU 只有一条当前余额记录。
- 余额可以从不可变流水完整重建。
- 业务过账接口使用数据库事务、行锁和幂等键。
- 已过账流水不可编辑或删除，纠错通过反向流水完成。

## 6. 核心数据库表

| 领域 | 核心表 | 关键约束 |
|---|---|---|
| 组织 | `organizations`、`memberships` | 成员在组织内唯一 |
| 范围 | `warehouses` | 仓库编码在组织内唯一；类型、国家、时区、收发能力与启停状态 |
| 商品 | `products`、`skus`、`product_images` | SKU 在组织内唯一 |
| 采购 | `suppliers`、`purchase_orders`、`purchase_order_lines` | 单号在组织内唯一 |
| 收货 | `receipts`、`receipt_lines` | 收货数量不得超过未收数量 |
| 库存 | `stock_balances`、`stock_ledger` | 仓库 + SKU 唯一余额 |
| 调拨 | `stock_transfers`、`stock_transfer_lines` | 来源仓与目标仓不同；发出/收货幂等键在组织内唯一 |
| 补货 | `replenishment_policies` | 组织 + 仓库 + SKU 唯一策略；周期、MOQ、整箱数均为正数 |
| 订单 | `sales_orders`、`sales_order_lines`、`stock_reservations` | 订单号在组织内唯一 |
| 出库 | `shipments`、`shipment_lines` | 同一订单不可重复出库 |
| 退货 | `return_orders`、`return_lines`、`return_receipts` | 累计退货不得超过已出库 |
| 监控 | `competitor_products`、`competitor_snapshots` | 商品 + 快照时间唯一；提醒当前由计算结果呈现，未单独持久化告警表 |
| 治理 | `audit_logs`、`local_imports`、各过账单据的幂等键 | 幂等键在组织内唯一；迁移报告不可变 |

组织作用域主表包含 `organization_id`；明细表通过父单据或商品关系继承组织范围。实体使用 UUID 主键、创建时间和更新时间，过账单据按业务需要记录操作人、幂等键与审计日志。

## 7. 权限模型

默认角色：

- **系统管理员**：成员、权限、全部业务和系统设置。
- **运营主管**：商品、采购、仓库、订单、竞品和报表，可审批调整。
- **采购员**：供应商、采购单、收货和补货策略，不可修改权限。
- **仓库员**：收货、库存调整、调拨、一键/分步出库和退货。
- **只读用户**：只读业务数据和审计日志。

当前权限同时检查动作和组织范围；仓库级成员范围是后续扩展，尚不能作为已实现能力对外承诺。

## 8. API 规则

- 统一前缀：`/api/`，所有路由保留末尾斜杠。
- 列表接口必须按当前组织隔离，禁止依赖前端传入组织 ID 作为唯一保护。
- 过账接口使用动作端点，例如：
  - `POST /api/purchase-orders/{id}/submit/`
  - `POST /api/receipts/`
  - `POST /api/stock-balances/adjust/`
  - `POST /api/stock-transfers/{id}/dispatch/`
  - `POST /api/stock-transfers/{id}/receive/`
  - `POST /api/stock-transfers/{id}/cancel/`
  - `GET /api/replenishment/recommendations/?warehouse={warehouse_uuid}`
  - `POST /api/orders/{id}/confirm-and-ship/`
  - `POST /api/orders/{id}/confirm/`
  - `POST /api/orders/{id}/allocate/`
  - `POST /api/orders/{id}/start-picking/`
  - `POST /api/orders/{id}/verify/`
  - `POST /api/orders/{id}/ship/`
  - `POST /api/orders/{id}/cancel/`
  - `POST /api/returns/receive-from-order/`
  - `POST /api/returns/{id}/receive/`
  - `POST /api/returns/{id}/reject/`
  - `POST /api/competitor-snapshots/quick-sales/`
  - `POST /api/local-imports/validate/`
  - `POST /api/local-imports/commit/`
- 收货、库存调整、调拨发出/收货、锁库、出库、退货收货和本机迁移等过账动作在请求体中接受 `idempotency_key`；同键不同业务内容会被拒绝。
- 错误返回用户可读中文消息和字段级校验信息；稳定业务错误码仍是后续增强。
- 成本、权限、过账和状态变化必须写审计日志。

普通资源集合还包括 `/api/warehouses/`、`/api/stock-transfers/`、`/api/replenishment-policies/`、`/api/competitors/` 和 `/api/competitor-snapshots/`。完整端点清单见 `backend/README.md`。

## 9. 单机数据迁移

1. 每台旧设备先导出完整 JSON；不要直接把不同设备的库存余额相加。
2. 指定一台设备为库存权威来源，逐 SKU 核对在库、锁定和在途，并导入一个尚无业务数据的新组织；锁定库存、开放采购、未出库订单和未完成退货会阻止切换。
3. 服务端接受 v5 与 v6 JSON；预检统计商品、SKU、竞品、快照、期初库存行，阻止版本错误、重复 SKU 和混入已有业务库。
4. 旧在库通过服务端库存服务生成“期初库存”流水，不直接写余额表。
5. 商品、SKU、供应商、可共享的 HTTPS 图片、竞品和快照在同一事务中导入，并保存本机 ID 到 UUID 的映射和审计报告。
6. v6 增加多仓、调拨和补货参数，但当前服务端导入仍只把期初库存写入用户选定的一个目标仓；不会导入本机仓库结构、调拨和补货参数。
7. 当前版本不把已闭环的旧采购、订单、退货和旧流水导入成可继续过账的单据；这些历史完整保留在原 JSON 中，待未来只读归档导入。
8. 本地 Data URL 与非 HTTPS 图片列入待补图警告；文件上传端点尚未实现。
9. 原浏览器数据不会自动删除。正式切换前仍需人工核对在途、锁定和历史单据。

## 10. 验收标准

- 两个不同账号能看到同一组织数据，并按组织角色限制写操作。
- 商品、图片、快照数量与迁移报告一致。
- 在途和在库逐 SKU 对账差异为零。
- 仓库切换后采购、在库、订单、调拨和补货建议不会串仓；调拨发出、收货和在途撤回均可由流水重建。
- 重复收货、重复出库请求不会重复过账。
- 一键出库遇到任一 SKU 库存不足时整单回滚，不留下锁定或出库流水。
- 补货建议能解释采购周期来源、7/15/30 日速度、库存位置、建议量、提醒等级和信心度。
- 竞品存在基线快照后，只提交累计销量即可生成继承其他字段的新快照。
- 库存余额能从流水重建并得到相同结果。
- 成本和库存调整在审计日志中包含操作人、原因和前后值。
- 数据库与本地媒体卷完成一次真实备份和恢复演练。
- GitHub Pages 本机模式继续可用，且不会误连团队数据库；验收人员明确知道它不等于联网团队版。
- 在目标服务器上完成域名、HTTPS、PostgreSQL、备份恢复和至少两个账号的真实协作验收后，才可声明正式联网部署完成。
