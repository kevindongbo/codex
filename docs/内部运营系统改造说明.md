# 内部运营系统改造说明

## 一、系统定位

本系统仅供内部成员使用，所有账号均属于同一个内部组织。系统不提供成员自行创建组织、加入组织或分组的入口。

## 二、账号与权限

### 主账号

- 主账号即 Django 超级管理员账号，是唯一的系统所有者。
- 可在“账号与权限”中创建、修改、停用子账号，并逐项分配权限。
- 可查看全部业务数据和审计记录。
- 可通过已配置的验证邮箱完成登录验证码验证、修改密码和找回密码。

### 子账号

- 由主账号创建后，直接使用“账号名 + 密码”登录。
- 不会看到创建组织、选择组织或分组的流程。
- 没有修改本人密码、找回密码、管理成员或修改主账号的入口。
- 仅能使用主账号在“账号与权限”中分配的业务能力。

可分配的业务能力包括：基础查看、商品与竞品、仓储、采购、订单、补货、数据导入和审计查看。

## 三、数据与同步

- 所有业务数据以服务器数据库为唯一来源。
- 浏览器每 3 秒检查一次数据版本；有更新时自动重新加载页面数据。
- 账号创建、停用及业务数据的新增、修改、删除都会推动数据版本更新。

## 四、资料录入规则

为保证日常录入不被不完整资料阻塞，以下记录允许先保存为草稿：商品、SKU、仓库、供应商、采购单、销售订单和竞品资料。

- 缺少名称、编码等基础信息时，系统会生成“待完善”占位内容或临时编号。
- 草稿可以后续补充和编辑。
- 会影响真实库存或履约结果的动作仍必须完整：商品启用、采购单提交、收货、库存调整、调拨发出/收货、订单确认/出库、退货收货。

这条边界避免“信息没填完就无法保存”，同时避免不完整资料直接影响库存与订单。

## 五、界面与后台

- 前台所有业务页面使用中文。
- Django 管理后台使用中文标题和中文业务模型名称。
- 默认的“用户、组、组织、成员关系”后台入口被隐藏，避免与系统内“账号与权限”形成两套管理方式。

## 六、上线前配置

- 必须配置生产数据库、`DJANGO_SECRET_KEY` 和域名相关安全配置。
- 邮箱验证码功能需要配置 SMTP。未配置时不会强制主账号进行邮箱验证，防止无法登录。
- 上线前先备份数据库与媒体文件，执行迁移后检查 `/api/health/`、主账号登录、子账号登录和页面自动刷新。
